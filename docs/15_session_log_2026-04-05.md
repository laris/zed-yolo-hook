## 4. ExitPlanMode Fix

### Option-Aware Send

`build_tool_outcome()` and `build_plan_outcome()` in `tool_authorization.rs`:
- Regular tools: sends configured `tool_option` (allow / allow_always)
- ExitPlanMode: sends configured `plan_option` (acceptEdits / bypassPermissions / default / plan)

### Detection

Reads first `PermissionOption`'s option_id from the `WaitingForConfirmation` entry:
- If `"allow_always"`, `"allow"`, `"reject"` → regular tool
- If `"bypassPermissions"`, `"acceptEdits"`, `"default"`, `"plan"` → ExitPlanMode
- Fallback: if pointer validation fails, assumes regular tool

### PermissionOptions Discriminant

Discovered: `PermissionOptions::Flat` = discriminant 0, but some sessions use discriminant 0x4 (Dropdown variant). Plan detection correctly skips non-Flat variants.

## 6. Restored Session Gap Discovery

### The Problem

"nixos" workspace showed "Awaiting Confirmation" dialog but hook logged zero activity for that session. All other workspaces (sessions `[s:2d00]`, `[s:3300]`) worked perfectly (80+ approvals, 0 misses).

### Root Cause Analysis

Two code paths for tool calls in Zed:

| Path | Source | Calls hook? |
|------|--------|-------------|
| `request_permission` → `request_tool_call_authorization` | Fresh tool calls | Yes |
| `session_notification` → `handle_session_update` → `upsert_tool_call` | Restored/resumed sessions | No |

The "nixos" session was restored from a previous Zed run. Tool calls arrived via the second path, bypassing `request_tool_call_authorization` entirely.

### Verification Steps

1. Checked all session tags: only `[s:2d00]` and `[s:3300]` — "nixos" session invisible
2. Clicked "Always Allow" manually → no new log entry (confirms hook never intercepted)
3. Compared ACP v0.24.2 vs v0.25.0 → identical `canUseTool()` flow (ruled out version issue)
4. Found two code paths in Zed source (`acp.rs:1445` vs `acp.rs:1584`)

## 7. Three-Approach Stale Scanner

### Approach 1: Hook `upsert_tool_call_inner`

Registers AcpThread pointer on every tool call insertion via lock-free atomic array. Ensures the scanner knows about ALL AcpThread instances.

Symbol found: `_RNvMsn_CseS5viS5XUxq_10acp_threadNtB5_9AcpThread22upsert_tool_call_inner`

### Approach 2: Hook `handle_session_update`

Registers AcpThread pointer from the session_notification path.

Symbol found: `_RNvMsn_CseS5viS5XUxq_10acp_threadNtB5_9AcpThread21handle_session_update`

### Approach 3: Periodic Stale Scanner

Background thread (2s interval) scans all known AcpThread entries for `WaitingForConfirmation` and auto-approves. Tracks attempted entries to avoid infinite retry on consumed senders.

### Initial Results (before deadlock fix)

- Approach 1 (upsert_hook) won the race — approved entries before tool_authorization fired
- Approach 2 (session_update_hook) found same entries ~50ms later (send failed, receiver dropped)
- Approach 3 (stale_scanner) also found entries (send failed, already consumed)
- **Session `[s:b900]` (nixos) was being auto-approved!**

## 8. Mutex Deadlock Bug

### Symptom

Zed Preview hung at 09:21. Three `.hang` reports in `/Library/Logs/DiagnosticReports/`.

### Root Cause

`scan_and_approve()` was called from both Frida interceptor context (on Zed's threads) and the stale_scanner background thread. Both took `Mutex<HashSet>` locks. Frida interceptor + Mutex = deadlock.

Hang report showed:
```
Thread 0xb13d8d    Thread name "yolo-stale-scanner"    11 samples (1-11)
  11  ??? (libzed_yolo_hook.dylib + 470392)
```

Main thread: `_sigtramp` (suspended by signal).

### Fix

1. Removed all `Mutex` from interceptor paths — upsert_hook and session_update_hook `on_leave` now do nothing (only `on_enter` registers thread via lock-free atomic)
2. AcpThread registration uses fixed-size `[AtomicU64; 64]` array — zero Mutex
3. Scanner thread uses `thread_local!` for attempted-entries tracking — no shared lock
4. Architecture: interceptor hooks (lock-free) → register thread → scanner thread (sole accessor of mutable state)

### Verification

After fix: no hang, 3 AcpThread instances registered, approvals working on `[s:db00]` and `[s:5e00]`, stale scanner correctly skips consumed entries.

## 10. Known Remaining Issues

1. **ExitPlanMode detection incomplete for Dropdown PermissionOptions** — discriminant 0x4 (seen on some sessions) is not Flat. Plan detection skips these. The option_id reading only works for Flat variant.

2. **Stale scanner latency** — 2-second scan interval means restored session tool calls may show the dialog for up to 2 seconds before auto-approval. Not instant like the primary hook.

3. **Log file date rollover** — rolling appender uses UTC dates for filenames, but timestamps are local. A log entry at `2026-04-05T07:00+08:00` goes to the `2026-04-04` file (UTC date). The verify health check looks for today's file and may miss it.

### Trigger

While verifying whether §6.3.1 of `zed-project-workspace/docs/15_refactor_workspace_management.md`
(the "always hook `is_lex = false` via Frida" pinning strategy) had been

`ForceNonLex` sibling-swap from §6.3 is still the only active pinning
mechanism. That confirmed the user's suspicion that the refactor might have
more drift from the design than the status table implied — so I audited
every phase.

### Findings: the v4 refactor has TWO classes of gap

#### Class 1 — known deferred (accurately documented)

- **Phase 3b** (Frida `is_lex` detour). Doc 15's own status table already
  marks this ⏸ deferred and explains the gating requirement (per-Zed-version
  symbol verification).
- **Phase 4c** (file watcher). Marked ⏸ deferred. `notify` crate + FSEvents
  integration not started.
- **Phase 4d** (wid-stable folder rename migration). Marked ⏸ deferred.

These are honest "not yet, here's why" entries. No action needed on the
doc front.

#### Class 2 — silently broken "✅ landed" claims (the real finding)

The uncommitted doc-15 status table marks these ✅ landed, but the library
code is **not wired into the production hook path**:

1. **`sync::decide` (Phase 4a, the 9-state decision matrix).**
   `zed-project-workspace/src/sync.rs:88` fully implements A1–A6, B, C, D
   with 9 unit tests. `grep 'sync::decide' zed-prj-workspace-hook/` → zero
   matches. Neither the hook nor the MCP calls it. It is dead code.

2. **`sidecar::write_sidecar` (Phase 4b write side).**
   `zed-project-workspace/src/sidecar.rs:26` writes sidecars with cache
   fallback, unit-tested. `grep 'write_sidecar' zed-prj-workspace-hook/` →
   zero matches. Nothing produces sidecars. The MCP
   `workspace_reconcile` tool (which is fully implemented at
   `zed-prj-workspace-mcp/src/main.rs:561-609` with all three strategies)
   has nothing to reconcile in practice.

3. **`identity::resolve_primary` runs only from MCP, not from the hook.**
   Triple-match validation, stale-project-name cleanup, wid-drift refresh
   do not fire on normal Zed DB writes — only when a user invokes an MCP
   tool.

4. **`pinning.strategy` config key (§6.3.1).** Designed as
   `"auto" | "frida_islex" | "force_non_lex" | "off"`. Actual
   `zed-prj-workspace-hook/src/config.rs` has only `enabled`, `log_level`,
   `sync_delay_ms`, `sync_cooldown_ms`, `discovery_cooldown_s`. No operator
   knob to choose pinning strategy or disable pinning without disabling the
   entire hook.

### What the hook actually does today

Every Zed DB write triggers this path (`zed-prj-workspace-hook/src/sync.rs:340-482`):

1. Read workspace row by `workspace_id`; rediscover if stale.
2. Read `.code-workspace` file.
3. Diff membership + order.
4. On any diff → silent rewrite of the user's `.code-workspace` (DB is
   always source of truth, file never contests).
5. Apply `PinAction` dispatch to correct `paths_order` in DB.

This is the A2 column of the SyncDecision matrix, applied universally.
A3 (divergence → sidecar) never fires. A4/A5/A6 (identity self-heal) never
fire.

### Impact

- **User data loss risk.** A user who hand-edits `.code-workspace` and then
  makes any change in Zed loses their edits with no warning. The sidecar
  safety net designed in §7.3 of doc 15 is infrastructure-complete but
  functionally inert.
- **Pinning strategy is stuck on fallback.** `ForceNonLex` sibling-swap
  works, but the `is_lex` detour was supposed to supersede it. Until 3b
  lands, there is no operator choice and no escape hatch (no `"off"`
  config value).
- **Self-healing is manual.** If `project_name` drifts, only MCP
  invocations can repair it. The event-driven hook — the high-frequency
  path — is blind to identity regressions.

### Roadmap re-evaluation (prioritized)

1. **Wire `sync::decide` into `do_event_driven_sync`** — rewrite hook
   `sync.rs:340-482` to build `IdentitySnapshot`/`DbState`/`FileState`,
   call `sync::decide`, dispatch A1/A3/A4/A5/A6 (A2 keeps today's
   rewrite). This single change converts 80% of the "dead library code"
   into live code. **~4 hours.**
2. **Add Frida `is_lex` detour** (Phase 3b) with auto-fallback to
   ForceNonLex when the symbol is absent. Log the chosen strategy at init.
   **~3 hours + per-version symbol verification.**
3. **Add `pinning.strategy` config key.** `"auto"` (3b-preferred) /
   `"force_non_lex"` / `"off"`. **~1 hour.**
4. **Add `workspace_fix_identity` MCP tool** so the hook's A5/A6 branches
   in step 1 can delegate destructive rename/demotion operations to the
   MCP (where isolation testing is easier). **~2 hours.**
5. **Phase 4c (file watcher)** — unblocked once step 1 lands, since the
   file-side trigger will then have a well-defined dispatch target.
6. **Phase 4d (wid-stable migration)** — depends on step 4.
