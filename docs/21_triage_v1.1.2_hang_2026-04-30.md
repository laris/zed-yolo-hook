# Triage: Zed v1.1.2 startup hang investigation (2026-04-30 → 2026-05-01)

> Status: **partially fixed**. The dominant cause (helper-subprocess spawn-storm)
> is eliminated by `process_role` multi-hop walk. A residual first-launch
> ~10–30 s window remains, attributed to dyld launch-closure rebuild + cold
> session restore — not to our hook's runtime work.
>
> Audience: future-me + reviewers of follow-up patches. This is a forensic
> record; nothing here describes "current code state". Code references are
> snapshots; offsets/SHAs/symbol hashes are dated.

---

## 1. TL;DR

After Zed Preview auto-updated from `v0.233.3` → `v1.1.2` on 2026-04-29, the
first patched launch hung for 30–53 s before becoming responsive. Two distinct
problems were tangled together:

1. **Helper-subprocess spawn-storm.** Zed v1.1.2's launch fans out ~9–11 sub-
   processes that all inherit the dylib (because `MacOS/zed` is patched).
   `zed-yolo-hook` had no helper-skip guard like the sibling
   `zed-prj-workspace-hook` got on 2026-04-23 (commit `2ee696e`). Every helper
   ran the full ctor — Frida init, 5 sequential `enumerate_symbols()` scans
   over a 354 MB binary, and a file-backed `HookRegistry::locked_register()`
   mutex — serialised on dyld + the cross-process lock.

2. **First-launch dyld launch-closure rebuild.** Whenever `cargo patch`
   rewrites `MacOS/zed`, dyld discards its cached launch closure for that
   binary and rebuilds on the next launch. On a 354 MB binary with our
   Frida-laden dylib added, this is unavoidably slow (≈ 10 s of dyld init
   *before* `main()` enters). Subsequent launches use the rebuilt closure
   and are fast.

**Status:**

| Issue | State | Evidence |
|---|---|---|
| Helper spawn-storm | Fixed by multi-hop `process_role::detect` | 23:22 / 23:23 / 03:15 launches log a single `DIAGNOSTIC: role=Primary` line each (was 11–12 in 20:18 launch under the one-hop guard) |
| First-launch closure rebuild | Inherent macOS behaviour, not hook code | 23:22 launch hung; 23:23 relaunch (same binary, same dylib) did not |
| Symbol resolution drift | Already absorbed by substring matchers | All 5 symbols found on v1.1.2 (Preview) and v1.0.0 (Stable) |
| AcpThread struct offset drift | None | `0xb0/0xb8/0x1c0/0x118/0x160/0x168/0x170` carry forward from v0.233.0 |
| Patcher silent inject failure | Not reproduced | `dylib-patcher` already bails on `insert-dylib` non-zero exit and on post-injection `verify()` |
| Hardened runtime / library validation | Not the cause (early hypothesis disproved) | Installed Preview is `flags=0x2(adhoc)` — no `runtime` flag |

- AcpThread offsets verified intact in
  `crates/acp_thread/src/acp_thread.rs:1038-1070` of upstream Zed at
  commit `a15106b`.

### 4.2 Phase 2 — first hang sample (14:17:16)

- 53 s spindump captured. PID 83035, fork-to-sample 65 s, main thread
  parked in `mach_msg`, all other threads quiescent for ~55 s.
- `cargo patch status` showed empty `Injected hooks (otool):` even
  though hook log showed 28 successful `YOLO mode ACTIVE` lines across
  many PIDs → suspected silent injection failure. Disproved later: that
  state was post-`cargo patch restore`, where `dylib-patcher` re-signs
  and the LC_LOAD_WEAK_DYLIB has been removed (only the 16 B signing
  delta remains).
- User ran `cargo patch restore` to revert before further investigation.

### 4.3 Phase 3 — three parallel research agents

Dispatched on 2026-04-30 to investigate three repos:

1. **dylib-kit** (`Explore` agent): claimed `insert-dylib` was silently
   succeeding without writing an `LC_LOAD_WEAK_DYLIB`. **Disproved:**
   the 16-byte file-size delta is from adhoc resigning during
   `restore()` (line 832 of `dylib-patcher/src/lib.rs`); the patcher's
   `verify()` in `lib.rs:495` would bail on missing inject.
2. **zed source** (`Explore` agent): claimed Zed v1.1.2's hardened
   runtime + commented-out `disable-library-validation` entitlement was
   blocking the dylib. **Disproved:** `script/bundle-mac` only applies
   `--options runtime` to *production* builds. The installed Preview is
   adhoc-signed (`codesign -dvvv` shows `flags=0x2(adhoc)` only — no
   `runtime` flag) so library validation is not active.
3. **zed-yolo-hook** (`Explore` agent): correctly identified missing
   helper-skip guard. **Confirmed.** This was the actual cause.

### 4.4 Phase 4 — port one-hop helper guard (correct in spirit, wrong in scope)

Mirrored `zed-prj-workspace-hook`'s `process_role` module verbatim:

```rust
let parent_exe = parent_executable_path(getppid())?;
if parent_exe == current_exe { return Helper; } else { return Primary; }
```

Result: 30 s hang remained; helper count unchanged.

Empirical observation from 20:18 log (§3.4): every helper's direct
parent is *not* a zed binary. The chain is

```
main MacOS/zed (ppid=1)
  └─ Node-like intermediate          ← not zed, doesn't load dylib
       └─ MacOS/zed grandchild       ← loads dylib, getppid() points at Node
```

So the one-hop check sees `parent_exe = Node`, classifies as Primary,
runs the full ctor.

### 4.5 Phase 5 — multi-hop walk (current state)

Walk up to `MAX_ANCESTORS = 5` ancestors using
`proc_pidinfo(PROC_PIDTBSDINFO)`. If *any* ancestor's exe equals
`current_exe`, classify as Helper.

Result: 23:22 / 23:23 / 03:15 launches show **1** Primary, no helpers
in log. Multi-hop fix is verified working.

### 4.6 Phase 6 — residual hang investigation

Despite multi-hop working, the user reported continued hang on first
post-patch launch. Investigation found:

- 23:22 launch hung; 23:23 relaunch (same binary, no re-patch) didn't.
- Hang trace file `[]` indicates Zed dispatcher hadn't scheduled tasks
  yet — i.e. hang occurred very early.
- Zed's "Rendered first frame" appears 1 s after start in every launch
  (no slowdown there).
- The hang detection at 23:22:39 fires 13 s into the launch, *during*
  the 11-second silent window between Zed's git-repo opens and prettier
  init (§3.3). Our hook is idle in this window — no calls intercepted
  yet.

Conclusion: residual hang is **first-launch dyld closure rebuild +
session restore**, not hook code. See §5.2.

## 5. Root cause analysis

### 5.1 Helper-subprocess spawn-storm (FIXED)

**Cause.** `MacOS/zed` is patched with `LC_LOAD_WEAK_DYLIB` referencing
our dylib. Every Zed self-fork inherits the dylib via the load command.
Zed v1.1.2's launch fans out 9–11 helpers (LSPs, file scanners,
crash-handler, ACP agent hosts via Node intermediates, etc.). Each
helper's `#[ctor]` ran the full init: `Gum::obtain()`, five
`enumerate_symbols()` walks across a 354 MB binary, and a file-backed
`HookRegistry::locked_register()` mutex.

**Compounded by.**
- Cross-process contention on the registry lock at
  `~/.config/dylib-hooks/zed-preview/registry.json`.
- Frida-gum static initialisers running in dyld's critical path.
- Cumulative `tracing-appender` traffic from every helper.

**Fix.** `process_role::detect()` walks the ancestor chain looking for
a zed-exe ancestor (§7).

**Evidence the fix works.** §3.4 — helper-init lines per launch dropped
from 11 to 1.

### 5.2 First-launch dyld closure rebuild (UNFIXED, INHERENT)

**Cause.** Whenever `cargo patch` rewrites `MacOS/zed`'s Mach-O load
commands and re-signs, dyld discards the cached launch closure for that
binary. The next launch must re-resolve every `LC_LOAD_DYLIB` /
`LC_LOAD_WEAK_DYLIB`, recompute rebase / bind tables, run every
constructor. With `libzed_yolo_hook.dylib` linked against frida-gum
(which has heavy static init) added to the load list, this is
unavoidably slow. Subsequent launches hit the rebuilt cache and are
fast.

**Evidence.**
- 23:22 (first after patch): `hang detected` 13 s in.
- 23:23 (relaunch, same binary): no hang detected, "Rendered first
  frame" in 1 s.
- 03:15 (Stable, separate launch closure): no hang detected.

**This is why the user saw two hang samples (14:17, 20:17) — both
were first-after-patch launches.** Both were captured before any
multi-hop relaunch warm-up.

### 5.3 Disproved hypotheses

| Hypothesis | Why disproved |
|---|---|
| Hardened runtime + library-validation entitlement | Installed Preview is adhoc-signed (`flags=0x2(adhoc)` only), no `runtime` flag. `script/bundle-mac --options runtime` applies only to production releases. |
| AcpThread struct layout drift | Offsets `0xb0/0xb8/0x1c0/0x118/0x160/0x168/0x170` carry forward; verified by 03:17:21 log line: `tool_authorization #1: matched v0.230.x entry[1] by ToolCallId, respond_tx=0x9322e2980` — successful match means offsets are correct. |
| `dylib-patcher` silent inject failure | `inject()` at `dylib-patcher/src/lib.rs:395-398` bails on `insert-dylib` non-zero exit; `patch()` at `:495-498` bails on `verify()` returning false. The "16-byte delta + empty otool" state observed was post-`restore`, not post-failed-inject. |
| Frida instrumentation overhead at runtime | Hooks installed in 53 ms total. After init, hook is idle until first hooked function call (22 s later in the 23:22 launch). The 11-s blocking window between first-frame and prettier-init is not in our code. |
| Quarantine attributes on dylib | `xattr libzed_yolo_hook.dylib` returns empty. |
| Code-signing verification (amfid) of injected dylib | Adhoc dylibs are accepted by amfid; no related `amfid` activity in the spindump. |

## 6. Quantitative findings worth keeping

### 6.1 Init cost on the primary

Hook `init_inner` measured wall-clock end-to-end:

| Launch | App | Total init | Per-symbol search |
|---|---|---|---|
| 23:22:26 | Preview v1.1.2 | 53 ms | 9-12 ms / symbol |
| 23:23:39 | Preview v1.1.2 | 65 ms | 9-13 ms / symbol |
| 03:15:29 | Stable v1.0.0 | 60 ms | 10-11 ms / symbol |

Each symbol search does `enumerate_exports()` first then falls back to
`enumerate_symbols()` (full symbol-table walk). For a 354 MB binary
with hundreds of thousands of symbols, ~10 ms per scan is reasonable.
5 scans × 10 ms ≈ 50 ms — matches measured.

These numbers are not a hot path. The 30–53 s hang spent < 0.2 % of its
time in our code.

### 6.2 push_entry call frequency on session restore

Log line `stale_scanner [s:9880]: found WaitingForConfirmation at entry[1661]`
(23:23:50) implies the AcpThread had at least 1662 entries at that point.
`push_entry` is called for each entry insertion. During session restore
this means our `push_entry_hook::Listener::on_enter` runs at least
1662 times in a tight burst, each call paying:

- Frida interceptor trampoline overhead (save context → call our fn →
  restore context → execute relocated prologue → return).
- Our `register_thread()` body: ≤ 64 atomic loads to check
  already-registered, return immediately if so.

Per-call cost is microseconds, total is milliseconds — not a hang
contributor on its own. Optimisation opportunity discussed in
plan doc 22.

### 6.3 Helper count per launch (Zed v1.1.2)

11 helper PIDs per launch in two waves visible in 20:18 log
(§3.4). Pre-fix every one of these ran full Frida init.

## 7. Fix details (current state of the code)

### 7.1 `src/process_role.rs` (new file)

- `ProcessRole::{Primary, Helper}` enum with `is_helper()` accessor.
- `detect()` walks up to `MAX_ANCESTORS = 5` ancestors via
  `parent_pid_of()` + `parent_executable_path()`.
- Match is by exact-equality of the ancestor's `proc_pidpath` against
  `std::env::current_exe()`.
- Failure modes (`ppid <= 1`, syscall failure at any hop, walk
  exhausted) all fail open as `Primary`.
- 6 unit tests, all pass under `cargo test --lib process_role`.

### 7.2 `src/lib.rs` integration

- `mod process_role;` added to module list.
- `init_inner` opens with:
  ```rust
  let role = process_role::detect();
  if role.is_helper() {
      let _ = std::io::Write::write_all(&mut std::io::stderr(), ...);
      return;
  }
  ```
  — single stderr line, no tracing, no Frida, no registry, no scanner.
- The existing `tracing::info!("config: ...")` line was promoted to
  `tracing::info!("DIAGNOSTIC: role={:?} ppid={} pid={} ...")` so the
  primary's role is forensically searchable.

### 7.3 What the fix did NOT change

- All hook listeners (`permission_decision`, `tool_authorization`,
  `upsert_hook`, `session_update_hook`, `push_entry_hook`).
- Offsets in `entry_scanner.rs` and `tool_authorization.rs` (still
  `0xb0/0xb8/0x1c0/0x118/0x160/0x168/0x170`).
- Symbol-search patterns in each hook module.
- `dylib-patcher` itself.

## 9. Open questions / known unknowns

1. **Why does Zed's main thread block for 11 s between first-frame and
   prettier-init on the 23:22 launch?** Either dyld is still resolving
   the freshly-rebuilt closure on a worker, or session-restore is
   reading 1662 ACP entries from SQLite synchronously on the main
   thread. Resolving this requires an instrumented `cargo run` of Zed
   itself — out of scope for the hook project.

2. **Does `cargo patch` need a "warm-up" launch?** The first-launch
